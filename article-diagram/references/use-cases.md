# Draw.io 使用示例

本文件提供常见 prompt、批量文章配图模式和任务拆解示例。只有在需要参考用户表达方式或多图输出模式时再读取。

## 单张图示例

### 流程图

```text
使用 $article-diagram 生成一个用户登录流程图，包含：
输入账号密码 -> 验证账号密码 -> 成功跳转首页 -> 失败提示错误并允许重试。
```

### 架构图

```text
使用 $article-diagram 生成一个微服务架构图，包含：
客户端、API 网关、用户服务、订单服务、认证服务、MySQL 主库、Redis、Kafka。
要求导出 PNG。
```

### 时序图

```text
使用 $article-diagram 生成一个订单创建时序图，展示：
客户端 -> API 网关 -> 订单服务 -> 库存服务 -> 支付服务 -> 返回结果。
```

## 多图文章模式

当用户给出一篇文章并要求“生成多张技术配图”时，推荐按下面方式处理：

```text
使用 $article-diagram 读取下面这篇文章，为文章生成多张合适的技术配图。要求：
1. 根据文章内容自动判断需要哪些配图
2. 所有配图放到同一个 draw.io 文件中
3. 主文件名与文章文件名一致
4. 每张子图作为一个 diagram page
5. 子图页面名采用英文小写中横线命名
6. 配图风格遵循 Editorial Clearline 规范
```

## 多图模式拆解建议

优先抽取这些结构：

- 总体架构
- 关键流程
- 调用链或时序过程
- 核心数据关系
- 异常处理或状态流转

避免把文章里的所有小知识点都画成一页。

## 页面命名建议

| 内容 | 推荐页面名 |
| --- | --- |
| 系统总览 | `system-overview` |
| 核心流程 | `core-flow` |
| 请求链路 | `request-sequence` |
| 数据模型 | `data-model` |
| 状态流转 | `state-transition` |

## 任务说明写法建议

- 尽量明确“节点是什么”和“节点之间是什么关系”
- 如果需要导出，直接写出格式
- 如果是给文章配图，最好给文章路径或正文
- 如果某些模块必须出现，直接点名，不要让模型猜
